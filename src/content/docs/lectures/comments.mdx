As your Python programs grow in complexity, documenting your code effectively becomes essential. Documentation refers to any notes or records that make your code easier to understand for yourself and others. The goal is to write **self-documenting code**â€”code that is so clear and well-structured that it serves as its own documentation. However, achieving perfectly self-documenting code is often impractical. Even if your code seems clear to you, expecting others to interpret it solely from the source is unrealistic. Therefore, supplementing your code with dedicated documentation is critical.

The simplest way to document your code is through **code comments**. A comment is text embedded in your source code that is not executed by the Python interpreter. Instead, it provides explanations for human readers, clarifying the purpose and logic of the code.

In Python, there are two primary types of comments: single-line comments and multi-line comments.

## Single-Line Comments

To create a single-line comment in Python, use the hash symbol `#`. Any text following the hash on the same line is treated as a comment and ignored by the interpreter. For example:

```python
# single_line_comment.py
import math

def main():
    pi = 3.14159265

    # Prompt the user for the radius of the circle
    radius = float(input("What is the radius of the circle? "))

    # Compute and print the area
    area = pi * math.pow(radius, 2)
    print(f"The area is: {area}")

if __name__ == "__main__":
    main()
```

Single-line comments apply only to the remainder of the line where they appear. If you need to write comments that span multiple lines, using single-line comments can become tedious, as each line must start with `#`:

```python title="Cumbersome Single-Line Comments"
# This is the first line of a comment
# This is the second line of the same comment
# And so on...
```

## Multi-Line Comments

For longer comments, Python uses triple quotes (either `"""` or `'''`) to create multi-line comments. Everything between the opening and closing triple quotes is considered part of the comment, allowing you to write explanations that span multiple lines without repeatedly adding `#`:

```python title="Multi-Line Comment Example"
"""This is a multi-line comment.
It spans several lines.
Use this format for long notes or explanations."""
```

Although Python does not have a distinct multi-line comment syntax like C++'s `/* */`, triple-quoted strings are commonly used for this purpose, especially for module-level or function-level documentation (often called **docstrings**). For example, you might use a docstring to document a function:

```python
def calculate_area(radius):
    """Calculate the area of a circle given its radius.
    
    Args:
        radius (float): The radius of the circle.
    
    Returns:
        float: The area of the circle.
    """
    return math.pi * radius ** 2
```

## Why Documentation Matters

Code comments are vital for ensuring your code is understandable, both to others and to your future self. For this course, you are expected to follow style guidelines that emphasize clear documentation through frequent, meaningful comments.

It might seem unnecessary to document code when working alone, but most significant software projects involve collaboration. Without proper documentation, your teammates will struggle to understand your code. Even for solo projects, you may find it difficult to recall the purpose of your code after some time. In professional team settings, thorough documentation is often a strict requirement, and poorly documented code may be rejected during peer reviews.

Beyond aiding others, comments benefit you directly. Writing comments encourages you to think critically about your code, which can help uncover logical errors, inconsistencies, or opportunities for improvement. Comments also serve as a roadmap for future development, reminding you of your thought process when you revisit the code later.

## Writing Comments During the Design Phase

A practical approach to effective documentation is to write comments as part of your design process, even before writing the actual code. This strategy helps you outline the structure of your program and clarify your intentions. For instance:

```python title="Commenting During Design"
# design_comments.py
import math

def main():
    # TODO: Prompt the user for the radius of the circle

    # TODO: Compute and print the area

if __name__ == "__main__":
    main()
```

By using comments like `# TODO:`, you create placeholders for tasks that need to be implemented. These comments act as reminders and help you focus on one task at a time. As you complete each task, you can replace the `TODO` comment with the corresponding code.

## Commenting Best Practices

When writing comments, adhere to the following best practices:

1. **Keep comments concise**: Avoid lengthy explanations for simple code. If a few lines of code require extensive comments, it may indicate that the code needs refactoring for clarity.
2. **Focus on clarity**: Write comments that are clear and relevant, providing just enough context to make the code understandable.
3. **Document for collaboration**: Assume others will read your code. Even solo projects benefit from good documentation, especially when revisiting code after a long period.
4. **Update comments**: As you modify your code, ensure your comments remain accurate. Outdated comments can be misleading and cause confusion.
5. **Use comments to explain why, not what**: Comments should explain the reasoning behind your code, not restate what the code does. The code itself should be self-explanatory.
6. **Avoid redundant comments**: Do not write comments that merely repeat the code. Comments should provide insights that are not immediately obvious from the code itself.

By consistently documenting your code with meaningful comments, you will enhance your workflow and improve your ability to collaborate effectively in team environments.
